# Research: GitHub Pages Automated Deployment

**Feature**: 007-github-pages-deployment
**Date**: 2026-03-27
**Status**: Complete - No NEEDS CLARIFICATION items

## Overview

This research document addresses technical decisions for the GitHub Pages deployment feature. Since the project already has a stable build pipeline and GitHub Pages is a well-established GitHub native feature, all decisions are based on existing project configuration.

## Key Decisions

### Decision 1: CI/CD Platform

**Choice**: GitHub Actions (GitHub's native CI/CD service)

**Rationale**:
- Already integrated with the repository (no additional setup)
- Free tier includes sufficient compute for our build
- Native integration with GitHub Pages deployment
- YAML-based configuration stored in version control
- Clear audit trail in GitHub interface

**Alternatives Considered**:
- CircleCI: Would require additional account/service, higher complexity
- Travis CI: Would require additional account/service, higher complexity
- GitLab CI: Would require moving entire project to GitLab
- Manual deployment scripts: Requires developer action, defeats automation purpose

**Verdict**: GitHub Actions is the optimal choice for a GitHub-hosted repository.

### Decision 2: Node.js Environment & Build

**Choice**: Use existing npm commands (`npm install` and `npm run build`)

**Rationale**:
- Build pipeline already defined and tested in project
- `npm run build` runs TypeScript compilation followed by Vite
- Build system is proven to work locally
- No modifications needed to existing package.json or configuration

**Alternatives Considered**:
- Custom bash script: Would duplicate existing npm configuration
- Node version management: Project already uses package.json (node version implicit from dependencies)

**Verdict**: Reuse existing build commands exactly as configured in package.json.

### Decision 3: Build Node Version

**Choice**: Use Node 18+ (latest available in GitHub Actions)

**Rationale**:
- package.json doesn't specify node version (uses implicit latest)
- TypeScript 5.9.3 and Vite 7.3.1 support Node 18+
- GitHub Actions provides latest stable Node versions

**Alternatives Considered**:
- Lock to specific Node version: Not currently enforced in project
- Use .nvmrc or .node-version: Can be added later if needed

**Verdict**: Use latest Node.js available in GitHub Actions runner, document in README for reproducibility.

### Decision 4: Artifact Handling & Caching

**Choice**:
- Build artifacts: Upload using GitHub Actions `upload-artifact` action
- Dependencies cache: Use `actions/setup-node` with npm cache built-in
- Final deployment: Use `actions/deploy-pages` action

**Rationale**:
- Official GitHub actions are maintained and optimized
- npm cache reduces build time (especially for repeated pushes)
- `deploy-pages` action handles GitHub Pages API authentication automatically

**Alternatives Considered**:
- Manual sftp/git push: More error-prone, harder to debug
- Third-party deployment actions: Less reliable than official GitHub actions

**Verdict**: Use official GitHub Actions for reliability and simplicity.

### Decision 5: Deployment Artifact (dist folder)

**Choice**: Deploy only the `dist/` folder generated by Vite

**Rationale**:
- Vite already outputs optimized production build to dist/
- Smaller deployment (only compiled assets, no source)
- Matches standard Vite/npm build process
- Requirement FR-004 specifies "dist folder as a static site"

**Alternatives Considered**:
- Deploy entire repository: Exposes source code, larger size
- Custom build output folder: Would require modifying existing build config

**Verdict**: Deploy dist/ folder exactly as Vite generates it.

### Decision 6: Build Verification (Failure Prevention)

**Choice**: Let `npm run build` exit with error code on failure; workflow respects non-zero exit codes

**Rationale**:
- GitHub Actions workflow automatically stops on step failure
- TypeScript compiler (`tsc`) exits with error on type errors
- Vite exits with error on build failure
- Clear error logging automatically available in Actions tab

**Alternatives Considered**:
- Add explicit validation step: Redundant (build already validates)
- Manual error checking: Harder to maintain

**Verdict**: Rely on build tool exit codes - proven pattern in CI/CD.

### Decision 7: Deployment Trigger Branch

**Choice**: Deploy only on push to `main` branch

**Rationale**:
- Requirement FR-001 specifies "main branch"
- Keeps production (GitHub Pages) separate from development branches
- Allows testing in feature branches without affecting live site
- Standard practice in CI/CD workflows

**Alternatives Considered**:
- Deploy all branches: Would pollute GitHub Pages with every branch
- Manual trigger: Defeats automation purpose
- Deploy on release tags: Would miss commits to main

**Verdict**: Trigger deployment on main branch only.

### Decision 8: Logging & Debugging

**Choice**: Rely on GitHub Actions default logging; no custom logging added

**Rationale**:
- GitHub Actions automatically logs all step outputs
- Each action (setup-node, build, deploy) logs relevant information
- Logs visible in GitHub Actions "Run" tab for each workflow execution
- Meets requirement FR-008 ("log deployment details")

**Alternatives Considered**:
- Add custom logging statements: Would clutter workflow, GitHub Actions already provides this
- Send logs to external service: Unnecessary complexity

**Verdict**: Use GitHub Actions native logging.

## Implementation Readiness

All decisions are based on:
1. **Existing project configuration**: package.json, tsconfig.json, vite.config.ts
2. **GitHub platform best practices**: Official actions, standard CI/CD patterns
3. **Miti constitution**: No modifications to app code, performance, or offline-first architecture

**No further research needed.** Ready for Phase 1 design and task generation.
